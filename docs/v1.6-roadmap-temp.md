# RBXMCP v1.6 Roadmap (Temp)

Temporary planning file for the next integration wave.
All three iterations are now implemented. Keep this file until you intentionally archive the v1.6 planning notes.

## Goal
Split the next MCP evolution into 3 practical iterations:
- Iteration 1: reduce AI confusion and token waste immediately
- Iteration 2: improve codebase navigation and dependency reasoning
- Iteration 3: add higher-level authoring flows that reduce round-trips

---

## Iteration 1: Bootstrap + Context Efficiency

Status: implemented

### Why first
This iteration gives the largest immediate gain for real chats:
- less token waste at the start of a session
- fewer blind searches
- clearer recovery from errors
- better project orientation for large games

### Deliverables
1. `rbx_get_project_summary(scope?: "all"|"scripts"|"ui", service?: string)`
- Returns compressed project map:
  - main services in use
  - script counts by service
  - top UI roots
  - likely entrypoints
  - likely hot spots
  - key remotes/modules counts
- Response should be LLM-oriented and compact.

2. `rbx_get_related_context(target: { path?: string[], symbol?: string, query?: string }, budgetTokens?: number)`
- Returns the smallest useful context bundle around a target:
  - target file/chunk
  - direct requires
  - callers/callees
  - nearby symbol definitions
  - optionally linked UI paths/remotes if found

3. `rbx_get_ui_summary(path: string[], forceRefresh?: boolean, maxAgeMs?: number)`
- Compact UI summary instead of full tree:
  - class histogram
  - interactive nodes
  - text-bearing nodes
  - layout primitives
  - likely controller bindings hints if inferable

4. `rbx_explain_error(code: string, details?: unknown)`
- Server-side error explainer.
- Must return:
  - what this error means
  - likely cause
  - recommended next call
  - whether retry is useful

### Capabilities update
Add to `/v1/agent/capabilities`:
- `recommendedNextStepByError`
- `bootstrapWorkflow`
- `preferredBootstrapCalls`

### Success criteria
- A new chat can orient itself with 1-3 requests.
- The model no longer needs to improvise how to recover from common API errors.
- Context bundles become meaningfully smaller than raw file reads.

## Iteration 2: Graph + Relevance Layer

Status: implemented

### Why second
Once bootstrap is efficient, the next bottleneck is navigation through large projects.
This iteration makes the MCP better at answering:
- where does this start
- what depends on this
- what remote/UI/script is connected to this

### Deliverables
1. `rbx_find_entrypoints(query?: string, service?: string, limit?: number)`
- Detect likely startup points:
  - boot scripts
  - UI controllers
  - server initializers
  - remote handlers
  - high fan-in modules

2. `rbx_find_remotes(query?: string, limit?: number)`
- Return remote map:
  - remote path
  - likely emitters
  - likely handlers
  - matched snippets

3. `rbx_rank_files_by_relevance(query: string, limit?: number)`
- Higher-level ranked retrieval.
- Should combine:
  - text hits
  - symbol hits
  - dependency hits
  - UI hits
  - remote hits
- Response should include `why` for each ranked result.

4. `rbx_get_changed_since(cursorOrTimestamp: string)`
- Return changed scripts/UI roots since a known point.
- Goal: avoid rereading stable project areas during long sessions.

5. `rbx_get_symbol_context(symbol: string, budgetTokens?: number)`
- Returns:
  - definition
  - references
  - local dependency neighborhood
  - best related snippets

### Internal work
- extend index for remote/event relationships
- add lightweight change journal for script/UI cache updates
- add file-level summaries to index metadata if needed

### Success criteria
- The model can find relevant files faster than plain text search.
- Multi-step reasoning over scripts/remotes/UI requires fewer manual calls.
- Long sessions stop re-reading the same stable files.

## Iteration 3: High-Level Authoring Flows

Status: implemented

### Why third
Only after retrieval is strong should we add expensive mutation abstractions.
Otherwise the model gets more write power without enough understanding.

### Deliverables
1. `rbx_apply_script_patch(path: string[], expectedHash: string, patch: string)`
- Patch-oriented alternative to full `newSource` replacement.
- Goal: reduce token usage and lower risk of unintended full-file rewrites.

2. `rbx_diff_script(path: string[], baseHash?: string)`
- Show concise diff for review/debug.
- Useful after edits and conflict recovery.

3. `rbx_clone_ui_subtree(path: string[], newParentPath: string[], newName?: string)`
- Efficient duplication for repeated UI structures.

4. `rbx_apply_ui_template(kind: string, targetPath: string[], options?: object)`
- High-level declarative UI generation for common patterns:
  - modal
  - shop grid
  - tab panel
  - list item
- Internally should compile into `apply_ui_batch`.

5. `rbx_validate_operation(kind: string, payload: object)`
- Dry-run validation before mutation.
- Should catch:
  - unsupported classes
  - out-of-root batch ops
  - stale versions/hashes when discoverable
  - malformed property payloads

### Success criteria
- Large edits require fewer round-trips.
- The model sends smaller mutation payloads.
- Users can inspect changes more safely before and after writes.

## Implementation Notes
- Keep everything additive.
- Prefer compact, LLM-oriented payloads over raw dumps.
- Each new tool should explicitly answer:
  - why this result is relevant
  - what next call is recommended
- Avoid adding mutation-heavy tools before retrieval quality justifies them.
